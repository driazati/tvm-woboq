187 * \brief Transform the high-level PrimFunc to a low-level version
188 * that can be used as an API function.
189 *
190 *
191 * The main task of this function is to create code to :
192 * - Map the values in the api_args to Var that is required by body.
193 * - Insert assertions to check type/value of the passed arguments.
194 *
195 * \param num_unpacked_args Number of arguments that
196 * are processed in plain form instead of packed form.
197 *
198 * \note
199 * The function signature have two cases
200 *
201 * let num_packed_args = len(api_args) - num_unpacked_args;
202 *
203 * if num_packed_args is zero:

205 *
206 * if num_packed_args is not zero:
207 * f(TVMArg* packed_args, int* packed_arg_type_ids, int num_packed_args,
208 * api_arg_k, api_arg_k+1, ... api_arg_n,
209 * TVMValue* out_ret_val, int* out_ret_tcode)
210 *
211 * where n == len(api_args), k == num_packed_args
212 *
213 * \return The pass.

409/*!
410 * \brief Compact the buffer access region by removing the buffer regions that are not accessed,
411 * i.e. narrowing the buffer shape and adjust the access region if necessary.
412 *
413 * Before narrowing, `B` is a `[16, 16]` buffer, but only a skinny vector `B[i, 0:16]` is accessed.
414 *
415 * \code
416 *
417 * for i in range(0, 16):
418 * with T.block():
419 * B = T.alloc_buffer(16, 16)
420 * for j in range(0, 16):
421 * B[i, j] = A[i, j] + 1
422 * for j in range(0, 16):
423 * C[i, j] = B[i, j] + 1
424 *
425 * \endcode
426 *
427 * This pass narrows the buffer shape and adjust its accessed region accordingly.
428 * In this particular case, because only a `1 * 16` vector of `B` is accessed,
429 * the pass narrows `B` to shape `[1, 16]`, and changes the access to `B[i, j]` to `B[0, j]`.
430 *
431 * \code
432 *
433 * for i in range(0, 16):
434 * with T.block():
435 * B = T.alloc_buffer(1, 16)
436 * for j in range(0, 16):
437 * B[0, j] = A[i, j] + 1
438 * for j in range(0, 16):
439 * C[i, j] = B[0, j] + 1
440 *
441 * \endcode
442 *
443 *
444 * \return The pass.
445 */
446TVM_DLL Pass CompactBufferAllocation();

531/*!
532 * \brief This pass transforms annotated loops into pipelined ones where producers and consumers
533 * are overlapped with the information provided in loop annotations, which enables optimization
534 * techniques like prefetching and pipeline parallelism.
535 *
536 * The pipeline scope consists of the direct children of the annotated loop (ignoring BlockRealize,
537 * Block, SeqStmt), and the number of children is denoted by `n` in the documentation.
538 *
539 * The following annotations are used to guide the loop transformation:
540 *
541 * 1) Loop annotation `software_pipeline_stage` defines the pipeline stage.
542 * An array of `n` integers, and each element should be in range [0, max_stage],
543 * where max_stage is the maximum (inclusive) stage.
544 * 2) Loop annotation `software_pipeline_order` defines the pipeline order.
545 * An array of `n` integers, a permutation of [0, 1, ..., num_components - 1];
546 * 3) Block annotation `double_buffer_scope` controls certain buffer sizes to allow decoupling of
547 * read/write dependency. It's an integer index of the write regions of the block.
548 *
549 * Every annotated loop is transformed into a loop with three blocks as its direct children:
550 *
551 * 1) Prologue block, where components whose stage is less than `max_stage` is executed;
552 *
553 * 2) Body block, where all the components are executed;
554 *
555 * 3) Epilogue block, where only components whose stage is greater than 0 will be executed.
556 * The execution order is controlled by the annotation `software_pipeline_order`,
557 * and thus could be different than the original order.
558 *
559 * Note: For nested software pipelines, the inner software pipeline will be generated first,
560 * which may affect the number of the direct children of the outer loop.
561 * In this case, the annotations for the outer software
562 * pipeline should include the result of the inner software pipeline,
563 * which is the three blocks as discussed above.
564 * Example:
565 *
566 * Before this pass, the TIR is:
567 *
568 * \code{.py}
569 * @T.prim_func
570 * def before_transform(A: T.Buffer[(16, 16), "float32"], C: T.Buffer[(16, 16), "float32"]) -> None:
571 * for tx in T.thread_binding(0, 16, thread="threadIdx.x"):
572 * for i in T.serial(0, 16,
573 * annotations={"software_pipeline_stage": [0, 1],
574 * "software_pipeline_order": [0, 1]}
575 * ):
576 * with T.block():
577 * T.reads(A[tx, i])
578 * T.writes(C[tx, i])
579 * B = T.alloc_buffer((16, 1), dtype="float32", scope="shared")
580 * with T.block("B"):
581 * T.reads(A[tx, i])
582 * T.writes(B[tx, 0])
583 * B[tx, 0] = A[tx, i] * T.float32(2)
584 * with T.block("C"):
585 * T.reads(B[tx, 0])
586 * T.writes(C[tx, i])
587 * C[tx, i] = B[tx, 0] + T.float32(1)
588 * \endcode
589 *
590 * The TIR above annotates the loop as a two-stage pipeline with no reordering.
591 * After applying this pass, the TIR is transformed into:
592 *
593 * \code{.py}
594 * @T.prim_func
595 * def after_transform(A: T.Buffer[(16, 16), "float32"], C: T.Buffer[(16, 16), "float32"]) -> None:
596 * for tx in T.thread_binding(0, 16, thread="threadIdx.x"):
597 * with T.block():
598 * T.reads([A[tx, 0:16]])
599 * T.writes([C[tx, 0:16]])
600 * B = T.alloc_buffer([2, 16, 1], dtype="float32", scope="shared")
601 * with T.block("prologue"):
602 * T.reads([A[tx, 0]])
603 * T.writes([B[0, tx, 0]])
604 * B[0, tx, 0] = A[tx, 0] * T.float32(2)
605 * with T.block("body"):
606 * T.reads([A[tx, 1:16], B[0:2, tx, 0]])
607 * T.writes([B[0:2, tx, 0], C[tx, 0:15]])
608 * for i in T.serial(0, 15):
609 * with T.block("B"):
610 * T.reads([A[tx, i + 1]])
611 * T.writes([B[(i + 1) % 2, tx, 0]])
612 * B[(i + 1) % 2, tx, 0] = A[tx, i + 1] * T.float32(2)
613 * with T.block("C"):
614 * T.reads([B[i % 2, tx, 0]])
615 * T.writes([C[tx, i]])
616 * C[tx, i] = B[i % 2, tx, 0] + T.float32(1)
617 * with T.block("epilogue"):
618 * T.reads([B[1, tx, 0]])
619 * T.writes([C[tx, 15]])
620 * C[tx, 15] = B[1, tx, 0] + T.float32(1)
621 * \endcode
622 *
623 * The original loop has two blocks, B and C, as its direct children. The loop annotations indicate
624 * that block B has stage == 0, order == 0, block C has stage == 1, order == 1. Therefore, block B
625 * should be executed in advance of block C by one iteration. The order 0 and 1 specifies the order
626 * of block B and C inside the body block inside the result TIR.
627 *
628 * \return The IR transform pass.
629 */
630TVM_DLL Pass InjectSoftwarePipeline();
